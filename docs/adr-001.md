## Premissas, Lacunas e Riscos 
- Premissas:
1. Uso atual de API REST
2. A infraestrutura atual suporta a provisão de um serviço de mensageria gerenciado.
3. A equipe de desenvolvimento possui conhecimento básico em padrões assíncronos.
4. A consistência eventual é aceitável para painéis operacionais (atraso máximo de 5 segundos). 

- Lacunas de informação:
1. Não temos o pico exato de requisições por segundo em um cenário de falha parcial para usar como métrica.
2. Custo aproximado de operação do serviço de mensageria escolhido com base no volume de eventos.

- Riscos identificados:
1. **Risco:** O acúmulo de mensagens na fila sem consumo adequado (Backpressure). 
    * **Impacto:** Irá gerar atraso na visualização dos dados. 
    * **Impacto:** Pode ter perda dos dados se o periodo de retenção expirar.
    * **Mitigação:** Configurar uma DLQ para análise de falhas é uma forma de mitigação.
    * **Mitigação:** Implementar monitoramento e alertas para tamanho de fila e análise de falhas.
2. **Risco:** Complexidade operacional aumentada.
    * **Impacto:** Dificuldade de debug e monitoramento distribuído.
    * **Mitigação:** Padronizar logs de correlação (Trace ID) entre Ingestão, Fila e Worker.


## 1. Contexto
- Cenário de negócio e técnico:

A plataforma GreenEnergy coleta telemetria de usinas solares e eólicas em tempo quase real. Esses dados alimentam painéis operacionais críticos e relatórios regulatórios obrigatórios. Recentemente, a expansão da carteira de usinas elevou drasticamente o volume de eventos.
- Problema que motivou a decisão:

A arquitetura atual (monolítica ou acoplada) apresenta pontos únicos de falha na ingestão. Picos de telemetria estão causando lentidão no banco de dados, afetando tanto a escrita quanto a leitura (relatórios). Não há histórico formal das decisões de desacoplamento, o que gera risco técnico (onboarding lento e decisões contraditórias).
- Restrições conhecidas:
    - Não há orçamento para reescrever todo o sistema (Legacy).
    - A latência de ponta a ponta não pode exceder 10 segundos para alertas críticos.
    - Conformidade regulatória exige que nenhum dado de telemetria seja perdido.


## 2. Decisão
Fica decidido adotar um padrão de Arquitetura Orientada a Eventos para o fluxo de ingestão, estruturado em três pilares:

* **Separação de Responsabilidades:** Divisão clara entre API de Ingestão (Write) e API de Consulta (Read).
* **Ingestão Assíncrona:** A API de Ingestão não escreverá diretamente no banco principal. Ela publicará eventos em uma Fila de Mensagens. Um serviço worker consumirá essa fila para persistência.
* **Replicação de Leitura:** O banco de dados principal (Write) terá um réplica dedicada para atender a API de Consulta e relatórios analíticos, evitando contenção de I/O.

## 3. Alternativas Consideradas
| Alternativa | Vantagens | Desvantagens | Motivo de descarte/aceite |
|---|---|---|---|
|**1. Escrita Direta no Banco (Síncrona)**   | Simples de implementar e debugar. Consistência imediata.  | Baixa resiliência a picos. O banco torna-se gargalo único. Falha no DB derruba a ingestão.  | **Descartada.** Não atende aos requisitos de resiliência e volume crescente.  |
|**2. Processamento em Lote (Batch)**   | Alta eficiência de escrita no banco. Menor custo de transação.  | Latência alta (minutos/horas). Inviável para painéis "quase real-time".  | **Descartada.** Viola o requisito de negócio de tempo quase real. |
|**3. Arquitetura Orientada a Eventos (Fila + Worker)**   | Alta resiliência (buffer contra picos). Desacoplamento entre ingestão e persistência. Escalabilidade horizontal  | Consistência eventual. Maior complexidade operacional (monitorar filas).  | **Aceite.** Equilibra resiliência, escalabilidade e latência aceitável. |

## 4. Consequências
- Impactos positivos:
* Aumento da disponibilidade da API de ingestão (o * banco pode cair temporariamente sem perder dados, pois a fila retém as mensagens).
* Melhora de performance nas consultas analíticas (réplica dedicada).
* Capacidade de escalar o processo de persistência independentemente da API.
- Impactos negativos:
* Introdução de consistência eventual (dados podem demorar alguns segundos para aparecerem no dashboard).
* Aumento da complexidade de infraestrutura (novo componente: Broker de Mensagens).
- Riscos introduzidos:
* Necessidade de implementar mecanismos de idempotência no consumer para evitar duplicidade em caso de reprocessamento.

## 5. Plano de Implementação (conceitual)
- Etapas de adoção:
1. Provisionamento do Broker de Mensagens e configuração de tópicos/filas.
2. Refatoração da API de Ingestão para publicar eventos (em vez de salvar no DB).
3. Criação do Serviço Worker (Consumer) para persistência no DB Write.
4. Configuração da Réplica de Leitura e redirecionamento da API de Consulta.
5. Implementação de Dashboards de monitoramento da fila (lag, throughput).
- Dependências:
* Aprovação de custo do serviço de mensageria.
* Janela de manutenção para deploy da nova versão da API.
- Indicadores de sucesso:
* Latência de ingestão estabilizada mesmo sob carga.
* Zero perda de mensagens durante testes de caos (falha simulada do DB).
* Tempo de resposta da API de Consulta reduzido em pelo menos 40%.
## 6. Revisão e Próximos Passos
- Pontos a revisar no futuro:
* Avaliar se o volume justifica a migração para um banco de dados de série temporal (Time-Series DB) no futuro.
* Revisar políticas de retenção da fila conforme o custo de armazenamento evolui.
- Condições para reavaliar esta ADR:
* Se o lag da fila ultrapassar consistentemente 1 minuto.
* Se o custo operacional do broker exceder 20% do budget de infraestrutura.
* Se novos requisitos de consistência forte forem exigidos por regulamentação.
